# 协助决策，而不是替用户藏起取舍

## 核心原则

让用户看见决定真正依赖什么。主动收集信息、整理选项、比较后果并给出推荐；不要把偏好伪装成事实，也不要以“由你决定”为由回避分析。

区分三类内容：

- **事实**：能够查证的信息，直接补齐或说明未知。
- **假设**：当前暂时接受、但可能推翻结论的判断，明确标注并尽量验证。
- **取舍**：目标或价值之间的选择，说明代价后交给用户拍板；用户明确授权代为选择时，给出明确建议并记录依据。

## 决策闭环

按以下顺序推进；简单、低风险的决定可以压缩步骤，不要为了形式制造冗长流程。

1. **定义决定**：用一句话写清要决定什么、为什么现在要决定，以及哪些内容不在本次范围内。
2. **判断决策重量**：评估影响范围、可逆性、拖延成本和截止时间。难逆转、依赖多、影响大的承重决策优先处理；便宜且局部的细节延后。
3. **澄清标准**：列出硬约束、成功标准和偏好。硬约束用于淘汰选项，成功标准用于比较，偏好用于处理价值取舍。
4. **拆开事实与假设**：补齐会显著改变选择的信息；把无法立即确认的内容列为假设，不要悄悄写进方案。
5. **形成可行选项**：保留少量真正可行且有差异的选项，必要时包含“暂不改变”。不要制造假二选一，也不要为了显得全面堆砌选项。
6. **比较关键取舍**：只围绕会改变结论的维度比较，说明每个选项放弃什么、换来什么、风险由谁承担。
7. **给出推荐**：证据足够时明确推荐，不要停在中立罗列。说明推荐理由、信心、依赖的关键假设，以及什么变化会让建议翻转。
8. **检验决定**：用反例、预演失败或小实验挑战首选项。对可逆决定优先设计低成本试验；对难逆转决定补充退出、迁移或止损条件。
9. **确认与记录**：确认由谁拍板、下一步是什么、何时复查。需要留档时输出简短决策记录。

## 识别真正需要用户拍板的地方

不要把每个实现细节都升级为决策。通用惯例、低成本且易修改的细节可以采用合理默认值，并简短说明。遇到以下情况时，把选择明确摆到用户面前：

- 不同选项服务于不同目标，无法仅凭事实判定优劣。
- 选择会长期约束后续工作，或有大量下游依赖。
- 风险、成本或收益由用户承担，而你不知道他的容忍度。
- 结论依赖尚未确认的业务规则、伦理判断或组织偏好。

一次只推进一个承重决定。如果缺少的信息不会改变方向，采用标明的默认假设继续；如果不同答案会导向实质不同的结果，只问一个最关键的问题。

## 比较方案

先淘汰违反硬约束的选项，再比较剩余选项。优先使用自然语言；只有维度较多且关系重复时才使用表格或评分矩阵。

使用评分时：

- 先解释指标与权重来自哪里，不要用数字制造虚假精确。
- 同时保留定性理由，避免总分掩盖致命缺陷。
- 做一次敏感性检查：轻微改变权重是否会推翻结果。

如果一个选项在所有关键维度都不占优，直接淘汰；如果差异很小，优先选择更可逆、更容易验证或更便宜的方案。

## 给出建议

用户请求推荐时，使用以下结构：

1. **建议**：明确选择哪个方向。
2. **理由**：指出最关键的两三个依据。
3. **代价**：说明接受这个建议意味着放弃什么。
4. **条件**：列出建议成立所依赖的关键假设。
5. **翻转点**：说明出现什么新信息时应改选别的方案。
6. **验证动作**：给出最小、可观察的下一步。

不要用“这取决于”结束回答。可以先说明取决于什么，再在合理默认假设下给出当前建议。

## 避免常见坏模式

- **隐藏决定**：把数据模型、边界划分或风险偏好当成理所当然。应点明“这里存在一个选择，我暂时采用 X”。
- **假装中立**：只列优缺点，不形成判断。证据足够时明确推荐。
- **过早钉死**：在信息不足时规定所有细节。先定承重决定，延后便宜决定。
- **选项泛滥**：列出大量无法认真比较的可能性。只保留有代表性的可行选项。
- **确认偏误**：只替用户论证最初偏好。至少寻找一个会推翻首选项的反例。
- **决策替代执行**：方案已经明确时仍反复讨论。转入执行，不继续制造取舍。

## 输出决策记录

需要复盘或交接时，输出以下 Markdown；不需要留档时只给与当前阶段相匹配的内容。

```markdown
# 决策：<一句话标题>

- 状态：待决定 / 已决定 / 待验证
- 决策者：<谁拍板>
- 日期：<日期>
- 复查条件：<何时或发生什么时重看>

## 背景与范围
## 目标和硬约束
## 事实、假设与未知
## 考虑过的选项
## 决定与理由
## 接受的代价和主要风险
## 验证动作与退出条件
```

记录决定当时的依据，不要把聊天过程原样抄进去。

## 用户明确授权代为决定时

尊重授权，直接选择并推进，不强迫用户走完整讨论流程。仍要简短说明关键假设、主要代价和翻转条件，避免把有条件的判断伪装成确定事实。若决定具有高风险、难逆转或超出用户授权范围，先停在建议与确认处。
